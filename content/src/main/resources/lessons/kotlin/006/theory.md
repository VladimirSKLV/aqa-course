# Kotlin Beginner 006. Классы, data class и архитектура тестового проекта

## Зачем это нужно в реальных проектах
На этом этапе вы переходите от «просто работает» к **устойчивым автотестам**, которые дают предсказуемый результат в CI.

В ежедневной работе AQA это означает:
- меньше flaky-падений;
- понятные отчёты при регрессе;
- быстрее разбор инцидентов;
- проще подключать новых инженеров к проекту.

## Теория
1. Формулируйте ожидаемое поведение как проверяемые правила (status, обязательные поля, бизнес-ограничения).
2. Разделяйте test logic и технику выполнения запросов/действий.
3. Используйте Kotlin-фичи (data class, коллекции, `when`, extension-функции), чтобы код оставался компактным и читаемым.
4. Любую нестабильность среды (время, порядок элементов, временные 5xx) обрабатывайте осознанно: retry, ожидания, нормализация данных.

## Пример
```kotlin
data class User(val id: Int, val name: String, val active: Boolean)

fun activeUsers(users: List<User>): List<User> =
    users.filter { it.active }.sortedBy { it.id }

fun main() {
    val users = listOf(
        User(2, "Mila", true),
        User(1, "Alex", false),
        User(3, "Nika", true)
    )

    val result = activeUsers(users)
    println(result.map { it.name }) // [Mila, Nika]
}
```

## Реальное применение
- Проверки response-списков (фильтрация, сортировка, агрегации).
- Сравнение фактических и ожидаемых данных по ключам.
- Построение reusable-утилит для повторяющихся проверок.

## Практический чек-лист

- Легко ли сменить окружение (`dev/stage/prod-like`) без правки теста?
- Понятно ли из лога, почему тест упал?
